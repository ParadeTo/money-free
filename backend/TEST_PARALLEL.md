# 并行数据获取测试指南

## 测试目的

验证并行处理功能是否正常工作，包括：
- API 速率限制是否有效
- 并发控制是否正常
- 数据准确性是否保持
- 错误处理是否健壮

## 测试步骤

### 1. 小批量初始化测试（10只股票）

测试初始化脚本的并行处理：

```bash
cd backend
npx ts-node src/scripts/init-index-stocks-klines.ts 10 0
```

预期结果：
- 约 2-3 分钟完成（串行需要 ~7 分钟）
- 显示并发进度信息
- 显示实时速率（约 20-30 只/分钟）
- 无 API 错误

### 2. 增量更新测试（20只股票）

测试增量更新脚本的并行处理：

```bash
cd backend
npx ts-node src/scripts/incremental-update-latest.ts 20 0 --index-only
```

预期结果：
- 约 1-2 分钟完成（大部分股票应该已是最新）
- 显示详细统计（更新/已最新/无数据/失败）
- 无 API 限流错误

### 3. 监控 API 调用频率

在测试期间，观察日志中是否有：
- `🚦 Rate limiter initialized: 100 requests/minute` - RateLimiter 已启用
- 无 `429 Too Many Requests` 错误
- 无 Tushare API 限流错误

### 4. 验证数据完整性

测试完成后，检查数据库：

```bash
cd backend
npx prisma studio
```

验证：
- K线数据已正确插入
- 技术指标已正确计算
- 日期范围正确
- 无重复数据

- 10 只股票：~7 分钟

### 并行处理（新）
- 10 只股票：~2-3 分钟
- 速率：~20-25 只/分钟
- 提升：2.3-3.5x

## 故障排查

### 问题：API 429 错误

**原因**：速率限制器配置过高

**解决**：
1. 编辑 `src/services/datasource/tushare.service.ts`
2. 将 `new RateLimiter(100)` 改为 `new RateLimiter(80)` 或更低
3. 重新运行测试

### 问题：并发数太高导致性能下降

**原因**：并发数超过最优值

**解决**：
1. 编辑脚本中的 `CONCURRENCY` 变量
2. 尝试 6、10、12 等不同值
3. 找到最佳平衡点

### 问题：数据库锁定错误

**原因**：SQLite 并发写入冲突

**解决**：
1. 降低并发数至 4-6
2. 或添加重试逻辑
